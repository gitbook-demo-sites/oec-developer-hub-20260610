# Developer journey

{% stepper %}
{% step %}
### Discover the right product surface

A developer lands on the portal, understands which API family applies to their workflow, and sees whether login is required before touching production paths.
{% endstep %}

{% step %}
### Request access and configure environments

They create or receive portal access, get sandbox credentials, and verify auth plus callback domains before writing production code.
{% endstep %}

{% step %}
### Implement against guides and reference

They move between [partner guides](https://app.gitbook.com/s/XSPACE_GUIDES/README) and [API reference](https://app.gitbook.com/s/XSPACE_API/README) instead of bouncing between PDFs, tickets, and legacy portals.
{% endstep %}

{% step %}
### Test and certify

They complete scenario validation, webhook checks, and release readiness steps with a clear promotion path from sandbox to production.
{% endstep %}

{% step %}
### Operate and get help

They use support docs and release updates for steady-state operation, while internal OEC teams see additional authenticated overlays when needed.
{% endstep %}
{% endstepper %}

## Success criteria

- The first useful page is reachable in under two clicks.
- Login is visible early, not buried in support docs.
- Public and authenticated content feel like one product.
- Support questions fall back to known escalation paths rather than tribal knowledge.
