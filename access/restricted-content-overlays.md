# Restricted content overlays

## Good candidates for restricted overlays

- internal launch readiness notes
- customer-specific implementation caveats
- support diagnostics and workarounds
- escalation owners and operational contacts
- internal-only release timing

## Keep these public by default

- onboarding and access basics
- authentication model
- testing guidance
- core event and error behavior
- release notes intended for partners

## Rule of thumb

Restrict content because it is sensitive or operationally specific, not because it is inconvenient to polish.
