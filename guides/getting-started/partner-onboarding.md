# Partner onboarding

{% stepper %}
{% step %}
### Confirm the API family

Identify which OEC product or workflow your integration belongs to and validate the business use case before credentials are issued.
{% endstep %}

{% step %}
### Request portal access

Use the visible login and access path to create an identity that can later carry partner-specific visibility when needed.
{% endstep %}

{% step %}
### Receive sandbox credentials

Provision sandbox credentials, callback domains, and any required tenant identifiers before implementation starts.
{% endstep %}

{% step %}
### Read the core technical pages

Start with [authentication and environments](https://app.gitbook.com/s/f1W2PfBZBd1K2O1uYgH8/authentication-and-environments) and [common integration patterns](../build-and-validate/common-integration-patterns.md).
{% endstep %}

{% step %}
### Align on launch criteria

Before development gets deep, confirm production promotion requirements and support ownership.
{% endstep %}
{% endstepper %}

{% hint style="info" %}
This page deliberately surfaces access early. When partner docs include restricted overlays, login is part of onboarding, not a separate support issue.
{% endhint %}
